# SENTINEL Pipeline Operations

## Purpose

This document records the current operational shape of the ingestion pipeline while the repository is in transition. It is not the final user guide. Its job is to capture decisions, defaults, and file roles so a clean step-by-step guide can be produced later.

For the full command inventory of executable stage runners, use:

- `docs/stage-runners.md`

## Current Pipeline Components

### Main event pipeline

- `scripts/run_pipeline.py`
- `scripts/pipeline_core.py`
- `scripts/ingest_rss.py`
- `scripts/ingest_newsapi.py`
- `scripts/normalize_articles.py`

This is the current default ingestion pipeline. It is still monolithic in
places, but the source-collection layer is now partially disaggregated and
`scripts/run_pipeline.py` should now be treated as the main fast-path entry
point.

Current responsibilities:

- orchestrate curated RSS and archive ingestion
- orchestrate NewsAPI ingestion when configured
- fetch DSCA and DEA items
- pre-filter articles
- classify and cluster events
- add AI analysis for high-salience new events
- write `data/events.json`
- write source audit output to `data/review/source_audit.json`

Source-collection responsibilities are now split as follows:

- `scripts/ingest_rss.py`
  RSS feeds plus WordPress archive collection
- `scripts/ingest_newsapi.py`
  NewsAPI query and normalization logic
- `scripts/normalize_articles.py`
  shared article-record shaping

Legacy compatibility note:

- `scripts/fetch_events.py`
  retired wrapper that forwards to `scripts/pipeline_core.py`

### GDELT ingestion

- `scripts/ingest_gdelt.py`

GDELT has been separated from the default pipeline. This was done to keep normal runs fast and to prevent the default pipeline from being dominated by a slower, heavier source.

Operational rule:

- GDELT is disabled by default in `scripts/pipeline_core.py`
- GDELT runs only if `--gdelt` is passed
- GDELT can also be run on its own through `scripts/ingest_gdelt.py`

### Historical ingestion planning

- `scripts/historical_ingest.py`

This is a separate planning/orchestration entry point for deep historical
archive work such as 2000+ coverage. It is not the same as the fast monitoring
pipeline and does not yet replace source-specific historical connectors.

## Current Outputs

### Core event store

- `data/events.json`

The live event store used by the current dashboard.

### Review artifacts

- `data/review/qa_report.json`
- `data/review/duplicate_candidates.json`
- `data/review/source_audit.json`

These are internal operational artifacts for analyst review and pipeline monitoring.

### Canonical event layer

- `data/canonical/events.json`
- `data/canonical/events.jsonl`
- `data/canonical/articles.json`
- `data/canonical/event_article_links.json`

Built by `scripts/pipeline/build_canonical_events.py`.

This is the first schema-aligned bridge from the live event store into the analyst-facing data model. It is currently assembled from `data/events.json`, not yet directly from article-level records.

The builder now also emits first-pass source/report linkage files derived from
each event's merged `source_all` and `url_all` arrays. This lets provenance
views name the specific reports attached to a clustered event even before the
upstream pipeline is fully article-native.

Older live events can be upgraded in place with:

- `scripts/pipeline/backfill_event_articles.py`

This backfills `source_article_ids` and `linked_reports` into `data/events.json`
from existing event-level source and link fields.

### Actor-coded canonical layer

- `data/canonical/events_actor_coded.json`
- `data/canonical/events_actor_coded.jsonl`
- `data/canonical/actor_mentions.json`

Built by `scripts/pipeline/code_actors.py`.

This is currently a heuristic scaffold, not the final actor-coding system. It upgrades generic labels like `executive`, `military`, `external`, and `oc_group` into more analysis-ready actor records and creates a flat actor-mentions table for event-actor analysis.

The current heuristic actor layer also reads:

- `config/actors/nsva_registry_seed.json`

That registry is a seed knowledge asset, not a complete actor database. It is intended to grow country by country and improve named non-state actor detection over time.

### Cleaned research export

- `data/cleaned/events_clean.json`
- `data/cleaned/events.csv`

Generated by `scripts/generate_clean_events.py`.

## Source Layer

### Curated feed definitions

- `scripts/rss_sources.py`

The curated source list now lives outside the retired `fetch_events.py` wrapper. This allows source maintenance without reopening the rest of the pipeline logic.

Feeds are grouped conceptually using metadata like:

- `official`
- `defense_specialist`
- `investigative_security`
- `human_rights_monitor`
- `general_news`
- `policy_analysis`

This metadata is not yet fully used downstream, but it is now preserved at ingestion and can later feed provenance displays, QA rules, and review prioritization.

## Analyst Edit Persistence

The analyst console now reads the safe template `data/review/edits.template.json` in fallback mode and the local-only `data/review/edits.local.json` in authenticated analyst mode, in addition to the review queue, QA report, duplicate candidates, and canonical actor-coded layer.

Current workflow:

1. Open `apps/analyst-console/` from the local repo server.
2. Choose a session role: `RA`, `Analyst`, or `Coordinator`.
3. Save local review drafts in the browser.
4. Export those drafts into `data/review/edits.local.json`.
5. Run `python3 scripts/review/apply_analyst_edits.py`.

This writes:

- `data/review/events_with_edits.json`
- `data/review/review_queue_with_edits.json`

The project still does not have a write-capable backend. This file-based edit workflow is the interim persistence layer.

## Local Analyst Backend

The repo now also includes a lightweight authenticated local backend:

- `python3 scripts/review/run_analyst_server.py`

It serves the analyst console and exposes:

- `POST /api/login`
- `POST /api/logout`
- `GET /api/session`
- `GET /api/edits`
- `POST /api/edits`

This backend uses:

- `data/review/users.local.json` for local credentials
- `data/review/edits.local.json` for persistent analyst edits

On every successful save, it reruns:

- `python3 scripts/review/apply_analyst_edits.py`

The analyst console will automatically detect the backend when served from this local server and switch from export/import mode to authenticated direct-save mode.

Recent analyst-console improvements:

- visible saved-edit notices after login and save
- a dedicated `History` tab backed by analyst edit records
- analyst query parsing for filters such as country, salience, actor, QA flags, duplicate flags, and edited state

## Operational Defaults

### Fast default run

Default command:

```bash
python3 scripts/run_pipeline.py
```

Current behavior:

- runs without GDELT
- attempts curated RSS and RSS-like feeds
- attempts NewsAPI if configured
- attempts DSCA and DEA scraping
- writes source audit output

### Opt-in GDELT run inside the main pipeline

```bash
python3 scripts/run_pipeline.py --gdelt
```

This should be treated as a slower enrichment mode, not the standard daily path.

### Standalone classification stage

```bash
python3 scripts/pipeline/classify_events.py
```

This runs the classification stage explicitly against filtered staged article
records instead of only through the top-level fast pipeline command.

### Standalone GDELT run

```bash
python3 scripts/ingest_gdelt.py
```

### Standalone historical GDELT run

```bash
python3 scripts/ingest_gdelt.py --historical --since 2025-01-01
```

## Source Audit

### File

- `data/review/source_audit.json`

### Purpose

The source audit exists to answer:

- which sources are returning articles
- which sources survive keyword filtering
- which sources actually generate events
- which sources produce high-salience events

This should be used to prune weak feeds and tune the source mix before doing more structural pipeline work.

## Important Implementation Notes

### RSS timeout protection

RSS fetching in `scripts/pipeline_core.py` now uses an explicit HTTP request with a timeout before parsing. This was added after the pipeline stalled on the Miami Herald feed. The intended behavior is now:

- one slow feed should fail fast
- the full pipeline should continue

### Source curation is now evidence-driven

Source cleanup decisions should use `data/review/source_audit.json`, not intuition alone.

Examples already observed in live runs:

- some direct RSS endpoints are dead or return 403/404
- some Google News site feeds produce raw items but no retained articles
- some NewsAPI domains are low-quality noise and should be blocked explicitly

When a source is consistently broken or low-yield, prefer one of:

- replace direct RSS with a site-scoped Google News feed
- remove the source from the default fast pipeline
- keep it only in a slower or manual workflow

### Recent live-run observations

After the cleanup pass on 2026-04-01, the fast default pipeline produced:

- 220 raw articles
- 74 filtered articles
- 13 clustered candidate events
- 1 newly added stored event in that specific rerun

Sources that looked consistently useful in those runs included:

- `Folha de S.Paulo`
- `Infodefensa`
- `InSight Crime`
- `Crisis Group`
- `El País América`
- `The Guardian LatAm`
- `El Tiempo Colombia`
- `El Faro`

The main remaining weak point is NewsAPI noise. Even after blocking several low-quality domains, NewsAPI still contributes many low-yield outlets and should be tightened further over time.

After the NewsAPI allowlist pass and normalization extraction later on 2026-04-01, a subsequent fast run produced:

- 205 raw articles
- 68 filtered articles
- 14 clustered candidate events
- 0 newly added stored events in that specific rerun

This reduced raw and filtered volume while preserving useful event yield, which is the intended effect of the allowlist approach.

At that point, the strongest remaining NewsAPI contributors were:

- `DW (English)`
- `PBS`

Most other retained NewsAPI domains were low-yield and should continue to be evaluated conservatively.

### Network behavior

The pipeline requires network access for live RSS, NewsAPI, DSCA, DEA, and optional GDELT. Runs inside a restricted environment may appear to succeed while returning zero live articles due to blocked DNS or outbound network access.

### Import layout

Some one-off ad hoc tests may need to be run from the `scripts/` directory because `pipeline_core.py` imports helper modules like `rss_sources` and `ingest_gdelt` as sibling modules.

## Published Review Metadata

`scripts/publish/publish_dashboard_data.py` now prefers
`data/review/events_with_edits.json` when it exists, so analyst review decisions flow
into the published layer without exposing private notes.

The public payload in `data/published/events_public.json` includes:

- `review_status`
- `review_priority`
- `human_validated`
- `provenance_summary.reviewed_by_human`

This gives the public dashboard enough information to show whether an event was
reviewed or validated by a human while keeping `analyst_reasoning`,
review-history details, and other private workflow data out of the published
artifact.
